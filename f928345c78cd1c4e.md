# Tangram Navigator

Tangram Navigator 是一个面向技术研究者和信息分析人员的结构化网络资源聚合与导航工具。项目定位于对特定领域的公开网络资源进行系统化梳理、分类归档与状态监测，帮助用户以较低的信息摩擦系数完成从“发现资源”到“访问资源”再到“评估资源可用性”的完整工作流。该项目不提供任何资源本身，也不对资源内容进行任何形式的代理、缓存或修改，仅作为公开链接的整理与呈现层。

目标用户包括网络安全研究人员、数据分析师、技术文档撰写者、信息采集工程师以及任何需要定期访问大量分散网络资源并维持其可用性感知的技术人员。项目通过可定制的分类索引、离线可用的资源清单、以及轻量级的可用性探测脚本，解决多源异构网络资源在长期使用过程中面临的链接失效、域名变更、分类混乱等实际问题。

## 功能概览

- **分类索引体系**：内置多层次分类标签，支持对资源按主题、区域、服务类型等维度进行标记与筛选，便于快速定位目标资源。

- **原始链接直出**：所有资源链接以原始字符串形式完整呈现，不附加任何追踪参数、跳转中间页或 HTML 包装，确保链接的纯净性与可复制性。

- **可用性探测集成**：提供基于 HTTP 状态码的链接存活检测脚本，支持定时任务或手动触发，帮助用户快速识别失效或异常的资源入口。

- **离线静态生成**：项目构建输出为纯静态 HTML 与 Markdown 文件，无需后端服务即可完整运行，适合内网部署或本地文件系统直接打开。

- **版本化资源快照**：每次资源列表更新均记录变更日志，支持按时间点回溯历史资源集合，便于追踪域名或服务路径的迁移变化。

- **自定义分类扩展**：用户可通过编辑配置文件新增分类标签与资源条目，无需修改核心代码，降低维护成本。

- **多格式导出**：支持将资源列表导出为 JSON、CSV 与纯文本格式，方便导入其他数据处理工具或进行二次分析。

## 应用场景

1. **技术研究信息采集**：研究人员在开展特定领域的网络服务分布研究时，可使用 Tangram Navigator 维护一份结构化的资源入口清单，并通过内置探测功能定期检查资源可用性，减少人工重复验证工作。

2. **文档与报告附录管理**：技术报告或安全分析文档常需附带大量参考链接。使用 Tangram Navigator 管理附录链接，可确保文档交付时所有链接均已通过可用性检查，并在链接变更时快速更新。

3. **团队知识库基础层**：团队可将该项目作为内部知识库的底层链接底座，将分散于 Wiki、文档、邮件中的外部资源统一汇总至 Tangram Navigator，再通过标签体系与团队分工进行匹配。

4. **日常巡检辅助工具**：运维或安全运营人员可将项目中集成的可用性探测脚本配置为定时任务，每日获取资源列表存活报告，快速定位异常服务入口。

5. **开源项目外部依赖索引**：开源项目维护者可利用此工具整理项目所依赖的第三方文档、镜像站、代码仓库等外部资源，便于新贡献者快速获取所需环境信息。

## 快速开始

以下步骤适用于 Linux / macOS / Windows WSL 环境，需提前安装 Git 与 Node.js 18+。

```bash
# 克隆代码仓库
git clone https://github.com/tangram-navigator/tangram-navigator.git
cd tangram-navigator

# 安装项目依赖
npm install

# 执行资源列表构建与可用性探测
npm run build
npm run probe
```

执行完成后，静态站点文件将输出至 `dist/` 目录，可用任意 HTTP 服务器（如 `npx serve dist`）或直接在浏览器中打开 `dist/index.html` 查看。

## 贡献指南

1. 在 GitHub 上 Fork 本仓库，并克隆至本地开发环境。建议在 `dev` 分支上进行修改，避免直接操作主分支。

2. 新增或修改资源条目时，仅编辑 `config/resources.json` 文件，遵循既定的 JSON Schema（参见 `docs/config-reference/schema.md`）。添加资源需提供 url、category 数组及备注说明，不得包含任何非公开或敏感信息。

3. 提交代码前，务必执行 `npm run test` 确保所有单元测试通过，并运行 `npm run build` 验证构建流程无报错。若新增前端样式或交互逻辑，需同步更新对应的单元测试。

4. 提交 Pull Request 时，请清晰描述变更目的与影响范围，并附上本地构建后的截图（如涉及 UI 变更）。PR 标题请遵循 `[类型] 简短描述` 格式（类型包括 feat / fix / docs / chore）。

5. 若发现资源链接失效或分类错误，亦可直接提交 Issue，标注 `[资源维护]` 前缀并提供失效链接的当前响应状态码，维护团队将定期处理。

## 常见问题

**Q：可用性探测显示某个链接为异常，但浏览器中可以正常打开，为什么？**

探测脚本默认仅检查 HTTP 状态码，不执行 JavaScript 重定向或页面渲染。若目标站点依赖前端跳转或需特定 User-Agent 头，可能返回非 200 状态码。建议在 `scripts/probe.js` 中自定义请求头或跟随重定向配置后重新探测。此外，部分站点对非浏览器请求进行限流，可适当降低探测并发数或增加单次超时时间。

**Q：项目构建后生成的静态页面能否直接放在 CDN 或对象存储上运行？**

可以。整个构建产物 `dist/` 目录包含完整的 HTML、CSS 与 JavaScript，所有资源路径均使用相对路径，无需后端服务。将 `dist/` 目录上传至任意支持静态托管的平台（如 AWS S3、Cloudflare R2、阿里云 OSS 或腾讯云 COS）并配置公开读取即可。探测功能由后端脚本独立执行，前端页面仅展示缓存结果，不影响静态托管。
